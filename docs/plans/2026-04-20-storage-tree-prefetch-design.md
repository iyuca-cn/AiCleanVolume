# 目录树后台预取与秒开展开设计

## 背景

当前目录树已经从“整棵树一次性加载”改为“按展开懒加载”，内存占用明显下降，但用户首次点击目录时仍会遇到两个问题：

1. 热点目录预取层数和命中率不够，首次点击仍可能需要等待。
2. 当前 `LoadDepth` 实现没有真正截断深层 JSON 解析，很多“只取一层”的请求实际上仍在从头递归读完整段 JSON，只是没有把更深层对象挂到 `Children`。

## 目标

1. 保持目录树的懒加载特性，不回到整棵树常驻内存的旧实现。
2. 在扫描完成后后台继续预取更可能被点击的前 5 层目录，让大多数首次点击达到“即点击即查看”。
3. 不阻塞“扫描完成”时机，主扫描仍应优先快速返回当前层。
4. 仅做会话内内存缓存，不允许落盘，但要尽量复用已经解析过的结果，避免同一路径重复从头解析。
5. 将预取范围和缓存规模控制在可预测上限内，避免再次引入过高内存占用。
6. 对未预取到的冷门目录保留兜底按需加载能力，但不再把“正在加载...”作为常态路径。

## 方案

### 预取策略

- 首次扫描仍只加载根节点和第一层，目录树继续以懒加载为基础。
- 扫描完成后立即启动后台预取任务，不阻塞 UI。
- 预取顺序按“更可能被点击”的目录优先：
  - 根节点下按体积排序的前 N 个目录优先；
  - 对已预取的目录，再递进预取其下按体积排序的前 N 个子目录；
  - 仅继续到限定层数，固定为 5 层。
- 除了静态热点预取，还要加入点击预测：
  - 用户展开某个目录后，立即把该目录下最可能继续点开的若干子目录加入高优先级预取队列；
  - 点击路径附近的预取优先级高于根层静态扩散。
- 这样可以优先覆盖大目录、热点目录和当前交互路径，而不是无差别补全整层。

### 缓存与展开行为

- 预取结果按目录路径和扫描参数做会话内缓存，缓存内容就是该目录的一层扫描结果。
- 用户点击展开时优先查缓存：
  - 若命中缓存，立即把子节点注入行模型并展开，达到秒开；
  - 若缓存未命中，再走兜底扫描。
- 预取完成后如果用户还没展开该目录，只更新缓存，不立即刷新整张表，避免 UI 抖动。
- 发起新扫描后，旧缓存与旧预取任务应整体失效，避免串数据。
- 缓存只保存在当前进程内，不写磁盘。

### 资源控制

- 预取深度固定为 5 层，每层热点目录数、缓存节点数需要有明确上限。
- 预取任务使用后台线程串行或小并发执行，避免和用户主动操作抢占过多资源。
- 命中缓存后可直接复用结果，不重复扫描同一路径。
- 扫描器需要真正支持深度截断：当达到 `LoadDepth` 阈值时，深层 `files` / `children` 直接跳过，不再继续递归解析 JSON。

### 失败与兜底

- 预取失败不弹窗，不打断用户，只记录日志。
- 用户点击一个尚未预取完成的目录时，仍允许兜底加载。
- 兜底加载失败时再按现有交互提示错误。

## 影响范围

- 主要修改 `src/AiCleanVolume.Desktop/MainWindow.cs`，接入后台预取调度、缓存命中、点击预测和扫描代际失效。
- 建议新增一个桌面层预取协调类，隔离缓存与任务调度逻辑，避免 `MainWindow` 继续膨胀。
- 需要调整 `src/AiCleanVolume.Desktop/ViewModels/StorageEntryRow.cs`，减少“正在加载...”占位的常态展示路径。
- 需要修正 `src/AiCleanVolume.Desktop/Services/FolderSizeRankerScanProvider.cs` 的深度截断逻辑，避免浅层扫描仍然从头递归解析完整 JSON。

## 验收

- 扫描完成后，后台自动开始预取前 5 层热点目录。
- 常见大目录首次展开时直接出现子节点，不再先看到 `正在加载...`。
- 用户展开某个目录后，其热点子目录会被优先预测并预取。
- 未命中缓存的冷门目录仍可展开，不影响功能正确性。
- 重复扫描新路径后不会混入旧扫描的预取结果。
- 同一路径在当前会话内不会重复从头解析同一层扫描结果。
- 解决交互延迟的同时，内存占用仍明显低于旧的整树加载实现。
